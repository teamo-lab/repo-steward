# Edward v0.2.0 → v0.3.0 — Discovery Quality Improvement

## Context

Product feedback from clawschool team on v0.2.0 results:

> **shufan**: 都是代码安全/稳健方面的，我们更关心功能上的，比如 skill 文件下发后用户能不能装成功。这种扫不出来。
>
> **xiaoke**: 粗看这些问题是真实存在的，且在代码安全性 & 代码后续可维护性上还是很有价值的；但是在产品功能层面上没有扫出来任何相关的 bug。

## Strategy change

| | v0.2.0 | v0.3.0 |
|---|---|---|
| Analysis approach | Single-pass code scan, dimension list | Multi-phase: read README → identify user flows → trace E2E → hunt functional bugs first |
| Required reading | None | README + entry points + critical flow trace |
| Output priority | Code health (security, dead code, type safety) | Functional bugs first, code health secondary |
| New task types | — | `functional_bug`, `flow_break`, `ux_gap`, `compat_risk`, `doc_drift` |
| Required field | — | `userImpact` (what the user sees when bug triggers) |
| Confidence floor | 0.6 | 0.65 |
| Agent budget | 10 turns / $2 / 10min | 40 turns / $5 / 20min |

## Why these are better

The v0.3.0 findings address what shufan and xiaoke asked for:

- **Finding #4** ("Retest can lower score"): Directly answers shufan's "skill 文件下发后用户能不能装成功" — a paid upgrade can decrease user's score, defeating the purpose of the upgrade.
- **Finding #9** (q7 PPT dead-end): A user pays for an upgrade they can never complete because the question is locked.
- **Finding #10** (referral self-abuse): Real fraud vector — user can game the free-upgrade system.
- **Finding #6** (888888 magic SMS): Production has a hardcoded backdoor.
- **Finding #1** (dashboard 500): Admin dashboard crashes on real production data.

These are user-facing, product-level bugs — not just "the code could be more idiomatic".
