## Functional findings (6)

### 1. [functional_bug] Dashboard summary crashes with closed-DB error when report token metrics are missing

**Confidence:** 97% &nbsp;&nbsp; **Risk:** high

In `_dashboard_summary_and_series`, the SQLite connection is closed at the end of the `finally` block (line ~1775), but the subsequent `for row in report_rows` loop at line 1891 calls `_load_report_score_counts(db)` and `db.execute(FILTERED_DONE_COUNT_SQL)` using that already-closed connection. Any admin dashboard load that encounters a Minimax-generated report with `total_tokens <= 0` throws `sqlite3.ProgrammingError: Cannot operate on a closed database` and returns HTTP 500. Fix by fetching score_counts and total_done_row before closing the DB, or by re-opening the connection inside the loop.

**User impact:** Ops/admin opens the dashboard and sees a blank page or 500 error. This silently fails with no data whenever any report lacks token-cost metrics.

**How to verify:** Call GET /api/{DASHBOARD_SLUG} on a server that has any report with report_provider='minimax' and report_total_tokens=0

**Code locations:**
- `app/main.py` line 1775
- `app/main.py` line 1891
- `app/main.py` line 1893

### 2. [flow_break] ack-installed endpoint transitions directly to 'completed', making 'installed_acked' a dead state and bypassing task enforcement on retest

**Confidence:** 95% &nbsp;&nbsp; **Risk:** medium

The docstring for `POST /api/upgrade/basic/tasks/{task_id}/ack-installed` says 'issued -> installed_acked' but the SQL sets `status='completed'` immediately. Because `_find_active_upgrade_task` only returns tasks with status in `{'issued','started','installed_acked','retest_submitted'}`, the task is invisible as 'active' when the agent submits the retest. This means the `upgrade_task_id` enforcement in `test_submit` (lines 2291-2309) is dead code and the retest submit proceeds without any task linkage check.

**User impact:** The upgrade completion event in task tracking fires before the retest is even submitted, corrupting analytics. An agent that installs skills but never retests will show a 'completed' task, misleading both the user and ops dashboard.

**How to verify:** Call ack-installed then immediately GET the task — status will be 'completed' not 'installed_acked'

**Code locations:**
- `app/main.py` line 3370
- `app/main.py` line 3415
- `app/main.py` line 2291

### 3. [flow_break] Retest submit via upgrade-task flow overwrites score without max-merge, so paid upgrades can lower the user's score

**Confidence:** 93% &nbsp;&nbsp; **Risk:** high

When an agent submits with `retest=true` through `POST /api/test/submit` (the new upgrade-task flow), `score_submission` is called on the freshly submitted answers and the result unconditionally overwrites the stored score. There is no `merge_retest()` call here — that function is only used in the now-unpaid legacy `/api/upgrade/basic` endpoint. If the agent regresses on any previously-correct question (e.g. q4 browser or q8 spreadsheet) the user ends up with a lower total than before they paid ¥19.9.

**User impact:** A user pays ¥19.9 for a basic upgrade, the agent retests but scores 0 on a question that was previously 10, and the user's displayed IQ drops. They paid for improvement and got a regression.

**How to verify:** Submit a first test with perfect q1 answer, then submit retest=True with q1 omitted/wrong

**Code locations:**
- `app/main.py` line 2256
- `app/main.py` line 2386
- `app/main.py` line 2391

### 4. [functional_bug] q9 video duration scorer accepts 3-30 seconds but the question spec and diagnosis hints specify 5-15 seconds

**Confidence:** 93% &nbsp;&nbsp; **Risk:** medium

In `_score_q9`, the full-score condition checks `3 <= duration_seconds <= 30`, but the question instructions, the `DIAGNOSE_ANSWER_HINTS` for q9, and the RETEST-SKILL template all say the video must be '5-15 秒'. An agent that submits a 3-second or 25-second video gets 10/10, while a user reading the spec would expect it to fail. Conversely, an agent that produces a valid 14-second video and is told by the spec it is correct could still score 6 if other fields are missing, creating inconsistent feedback.

**User impact:** Agents are rewarded for videos outside the stated 5-15s range (e.g. 3s or 25s), which undermines the test's validity. Diagnostic feedback will tell users to produce 5-15s videos, but the scorer silently accepts much shorter/longer ones.

**How to verify:** Submit q9 with duration_seconds=3 and all other fields valid — scorer returns 10/10

**Code locations:**

### 5. [flow_break] q7 (PPT) is simultaneously in LOCKED_QIDS and UPGRADE_SKILL_PACKAGES, creating an unresolvable upgrade dead-end

**Confidence:** 85% &nbsp;&nbsp; **Risk:** medium

`LOCKED_QIDS` in `repair.py` includes `'q7'`, which means `UPGRADEABLE_QIDS` in `main.py` excludes q7. Users cannot create an upgrade task for q7. However, `UPGRADE_SKILL_PACKAGES` in `main.py` has a full entry for q7 ('banana-slides'), and the `DIAGNOSE_ANSWER_HINTS` provides detailed q7 install guidance. When a user runs diagnosis and sees a q7 failure, the diagnose endpoint calls `_build_prebuilt_skills` which skips q7 (not in BASIC_QIDS), but the frontend's upgrade ability list still shows q7 as a visible option since `UPGRADE_ABILITY_LABELS` includes it. Users can see and attempt to select q7 in upgrade flows that then silently exclude it.

**User impact:** Users who fail q7 receive repair guidance with a 'banana-slides' install target, but clicking upgrade never includes q7 in the task. Users are confused about why PPT ability can't be upgraded despite seeing instructions for it.

**How to verify:** Fail q7, call /api/test/diagnose, confirm generatedSkills has no banana-slides despite diagnose hints pointing there

**Code locations:**
- `app/repair.py` line 40
- `app/main.py` line 277
- `app/main.py` line 327

### 6. [flow_break] Referral self-abuse: same browser can create two tokens and self-refer to unlock free upgrade

**Confidence:** 80% &nbsp;&nbsp; **Risk:** medium

`/api/referral/bind` only blocks `sharer_token == referee_token`. It does not check if both tokens belong to the same `visitor_id` in `visitor_tokens`. A user can: open two browser tabs, create token A (stored in cookie), clear cookies, create token B, then call `POST /api/referral/bind` with sharer=A, referee=B. After token B exists in tests, `/api/payment/create` with `plan_type='referral_free'` for token A succeeds, granting a free ability upgrade.

**User impact:** Users can obtain free upgrades worth one ability (≈¥9.9-¥19.9) by self-referring, undermining the viral referral mechanism and causing revenue leakage.

**How to verify:** Create two tokens from the same client, bind them as referral, claim referral_free payment

**Code locations:**
- `app/main.py` line 4486
- `app/main.py` line 4487
- `app/main.py` line 3746

## Code-health findings (4)

### 1. [security_fix] Legacy /api/upgrade/basic accepts score rewrites without verifying payment

**Confidence:** 97% &nbsp;&nbsp; **Risk:** high

The `POST /api/upgrade/basic` endpoint (the pre-task upgrade path still accessible at runtime) rescores any submitted question answers and max-merges them with the stored results, but performs zero payment verification. Any caller who knows a valid token can POST crafted perfect answers to silently elevate their score to 160+ IQ without paying ¥19.9. The endpoint's docstring even says '¥19.9 基础能力升级重测' but the implementation has no `payments` table check whatsoever.

**User impact:** A malicious user can bypass the payment wall and obtain artificially inflated scores and leaderboard positions for free, directly undermining the product's revenue and leaderboard integrity.

### 2. [security_fix] Magic SMS code 888888 is permanently valid in production — full authentication bypass

**Confidence:** 92% &nbsp;&nbsp; **Risk:** high

`MVP_VERIFY_CODE = '888888'` at line 3610 is hardcoded and checked unconditionally before any real code lookup. The comment says it's for development, but there's no guard based on `IS_DEV_ENVIRONMENT`. Anyone who knows any user's phone number (e.g. from a referral leak or social engineering) can log in as that user in production by entering 888888, bind the user's tokens to their account, and access their payment history and upgrade commands.

**User impact:** Any party who knows another user's Chinese phone number can immediately authenticate as that user in production, bind that user's test tokens to a new session, and gain access to their payment-linked upgrade commands.

### 3. [security_fix] Phone number cookie set without HttpOnly — phone number exposed to JavaScript

**Confidence:** 90% &nbsp;&nbsp; **Risk:** medium

In the `/api/login` handler, `clawschool_phone` is set with `samesite='Lax'` but without `httponly=True`. Any JavaScript running on the page (including third-party analytics or a future XSS vector) can read `document.cookie` and extract the user's phone number in plaintext. The `secure` flag is also absent, meaning the cookie could be transmitted over HTTP if the connection degrades.

**User impact:** If the site ever has an XSS vulnerability or loads third-party scripts that execute XSS, users' phone numbers are directly leaked. In China, phone numbers are personally identifiable under PIPL and leaking them is a regulatory risk.

---

### 4. [security_fix] Test tokens generated with non-CSPRNG random.choices — token space is predictable

**Confidence:** 88% &nbsp;&nbsp; **Risk:** medium

`_gen_token()` uses `random.choices()` from Python's standard `random` module, which is seeded from `os.urandom` at import time but is a Mersenne Twister — after observing ~624 outputs the state can be fully reconstructed, allowing future token prediction. An 8-character base-36 token has ~2.8 trillion combinations, but with a reconstructed PRNG state an attacker can predict every future token within milliseconds. Using `secrets.token_hex` or `secrets.token_urlsafe` would eliminate this risk.

**User impact:** An attacker who monitors token creation patterns could predict upcoming tokens, pre-claim test slots, hijack test submissions for other users, or submit fraudulent answers under predicted future tokens before the real user can.
